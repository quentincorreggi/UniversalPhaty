# UniversalPhaty Platform

This document describes the reusable prototyping workflow. It is
game-agnostic: every rule here should hold whether the repo hosts a
Marble Sorter, a platformer, a puzzle game, or anything else. Game-specific
architecture lives in `GAME.md`.

## Core Idea

One repo = one game. Each prototype lives on its own branch. Every push
to any branch auto-deploys to a unique versioned URL on GitHub Pages. A
landing page lists all shared prototypes, so teammates can browse and
play them in one click.

## Prototyping Workflow

### Step-by-step process for Claude

1. **Understand the mechanic.** Ask the user what they want to build.
2. **Align on the design.** Before writing any code, present a plain-
   language design brief covering the four dimensions in the
   [Alignment Phase](#alignment-phase). Propose concrete defaults and let
   the user confirm or adjust.
3. **Create a branch from `main`:**
   ```
   git checkout main && git checkout -b prototype/<slug>
   ```
   - Use the `prototype/` prefix. This is required for the prototype to
     appear on the library landing page.
   - `<slug>` is a short, lowercase, hyphenated name (max 40 chars).
4. **Implement the mechanic** following the patterns in `GAME.md`.
5. **Validate syntax.** Run `node --check` on each modified or new JS file.
6. **Commit** all changes with a descriptive message.
7. **Push:**
   ```
   git push -u origin prototype/<slug>
   ```
8. **Construct the preview URL** (see below). You can do this immediately
   — don't wait for the deploy.
9. **Share the URL** with the user along with a plain-language summary of
   what was built and how to try it.

### Safety Rules

- **NEVER** commit directly to `main`.
- Each prototype is isolated on its own branch.
- Keep prototypes self-contained — don't depend on other prototype
  branches.
- The game must remain playable — always verify the entry point loads
  without errors.

## Alignment Phase

Before coding any prototype or iteration, confirm alignment with the user
on four dimensions:

1. **Design** — Core rules, behavior, edge cases, interactions with
   existing mechanics.
2. **Interaction** — How the player triggers or uses the feature.
3. **Visuals & Feedback** — Colors, shapes, animations, particles, sound.
4. **Authoring / Editor** — How the mechanic shows up in whatever level
   editor or authoring surface the game provides.

Always propose specific defaults rather than asking open-ended questions.
The user should be able to approve with a single "go for it". Keep the
language plain and non-technical.

## Getting the Preview URL

Each push is auto-deployed to GitHub Pages via
`.github/workflows/deploy-branch.yml`. The URL includes the short commit
SHA so it is always unique — no caching issues.

**URL pattern:**
```
https://quentincorreggi.github.io/UniversalPhaty/<branch-name>--<short-sha>/
```

Branch names containing slashes are converted: `/` becomes `--`. Get the
short SHA with:
```
git rev-parse --short=8 HEAD
```

**IMPORTANT:** The SHA must be exactly 8 characters. Using 7 characters
(the git default) will produce a 404. Always pass `--short=8`.

Examples:
- `prototype/magnet-box` at commit `abc12345`
  → `https://quentincorreggi.github.io/UniversalPhaty/prototype--magnet-box--abc12345/`
- `claude/my-feature-abc123` at commit `deadbeef`
  → `https://quentincorreggi.github.io/UniversalPhaty/claude--my-feature-abc123--deadbeef/`

After pushing, the deploy takes 30-60 seconds. You can construct the URL
immediately without waiting.

Each new push to a branch **replaces** the previous versioned folder for
that branch, so only the latest deployment is kept on gh-pages per branch.

To verify the deploy completed:
```
gh run list --workflow=deploy-branch.yml --limit=3
```

## Prototype Library Landing Page

All shared prototypes are listed on the landing page at:
```
https://quentincorreggi.github.io/UniversalPhaty/
```

The landing page reads `manifest.json` (auto-generated by the deploy
workflow) and then fetches each branch's `prototype.json` (optional) for
metadata like human-readable name, description, and author.

**A prototype only appears on the landing page if it has a
`prototype.json` at the repo root of that branch.** Use `/share` to
generate one. Branches without `prototype.json` are still deployed and
playable at their direct URL, just not listed.

## Platform Conventions

These conventions apply to every game built on UniversalPhaty. They exist
to keep the codebase fast for Claude to edit and fast to iterate on.

- **No build step.** The entry point (`index.html`) loads directly in the
  browser. This keeps the edit → push → play loop measured in seconds.
- **No npm / frameworks / modules.** Vanilla JS only. Script tags in
  `index.html` in a deterministic order.
- **Small files.** If a file grows past ~700 lines, consider splitting.
  Claude edits more reliably against small, focused files.
- **Registry / plugin patterns.** New entities (box types, enemies,
  power-ups, whatever the game uses) should register themselves via a
  registry so that adding one new file + one script tag is enough. See
  `GAME.md` for the current game's registry contract.
- **Validate with `node --check`** before committing — it catches syntax
  errors without needing a browser.

## Adopting UniversalPhaty for a New Game

To use this repo as a starting point for a different game:

1. **Fork or "Use this template"** on GitHub.
2. **Keep** the following, unchanged:
   - `.github/workflows/deploy-branch.yml`
   - `landing.html`
   - `.claude/commands/{prototype,iterate,share}.md`
   - `PLATFORM.md` (this file)
   - `CLAUDE.md` (the thin pointer)
3. **Replace** the following with your game:
   - `index.html` (entry point)
   - `js/*` (game code)
   - `GAME.md` (describes your game's architecture to Claude)
4. **Update** user-visible strings and URLs to match your repo / game:
   - Repo name in `PLATFORM.md` URL examples
   - Repo name in `.claude/commands/share.md`
   - `<title>` and header text in `landing.html` (currently names the
     Marble Sorter game — change to your game's name)
   - `<title>` and any in-game header text in `index.html`
5. **First commit to `main`** a minimal playable skeleton (even just a
   blank canvas with one registered entity). Every prototype branch starts
   from `main`, so `main` must always be runnable.
6. **Write `GAME.md`** before asking Claude to add mechanics. The quality
   of that document determines how well Claude can modify the game.

A good `GAME.md` answers: What files exist and why? What is the registry
contract (so Claude can add new things)? What global state exists? What
invariants must be preserved?
